# AJTBD-промпты для Perplexity (бренд + сайт) — ADVANCED-пак

> **Что это и зачем.** Набор готовых промптов для Perplexity Deep Research, чтобы по сайту и бренду конкурента вытащить не маркетинг, а **работу клиента, силы переключения, незакрытые шаги и вектор движения** — в форме, готовой лечь в [досье](../examples/gong/dossier-skeleton.md) и [battlecard](../templates/battlecard.md).
>
> **⚠️ Это продвинутый слой.** Промпты намеренно плотные: внутри зашита механика Advanced JTBD (4 силы Moesta, desired outcome по Ulwick, job map, signal convergence). Если вы только начинаете — сначала пройдите [базовый внешний пак](perplexity-external-pack.md) (отзывы / найм / фандинг), а сюда возвращайтесь, когда нужна глубина именно по бренду и сайту.
>
> _Провенанс: пак закалён AJTBD-разбором (арена 3 школ — Замесин / Moesta / Ulwick). Ниша и роли в примерах обобщены до мульти-доменных._

## Плейсхолдеры

Перед прогоном подставьте свои значения:

| Плейсхолдер | Что это |
|-------------|---------|
| `{{КОНКУРЕНТ}}` | бренд конкурента |
| `{{САЙТ_URL}}` | его сайт |
| `{{НИША}}` | ваш рынок / категория (например: «SaaS для управления проектами», «студия ремонта под ключ», «сеть магазинов у дома», «производство упаковки») |
| `{{МОЙ_ПРОДУКТ}}` | ваш продукт (нужен только для финальной фразы вывода — но в промптах конкурента **не** сравнивайте «у нас vs у них», см. правило 4) |

## Общие правила разметки (действуют во ВСЕХ промптах)

Эти 4 правила — ядро метода. Они отделяют реальную работу клиента от витрины продавца. Каждый промпт ниже их наследует.

**1. Размечай источник каждого факта.**
- `[САЙТ/продавец]` — гипотеза продавца о клиенте (то, что написано на сайте).
- `[ГОЛОС КЛИЕНТА]` — цитата из отзыва / кейса / форума / интервью.

Силы и боль ценны именно во **второй** колонке. Сайт — это отполированный *pull* (притяжение нового): push (боль старого), anxiety (страхи внедрения) и habit (привычка) на нём почти стёрты. Их надо реконструировать и честно помечать `^[inferred]`. _^[inferred — ядро правки Moesta]_

**2. Не присваивай уровень Job (Big/Core/Small) и не клей номера механик.** Это сделает аналитик на разборе. Извлекай **наблюдаемое**: какую работу клиента решение убирает / сокращает / поднимает на уровень выше + есть ли артефакт-доказательство. Глагол с сайта = гипотеза уровня, не сам уровень. _^[inferred — veto Замесина: иначе модель клеит ярлыки на лозунги]_

**3. Любую «ценность» сворачивай в desired outcome клиента** по формуле: `[сократить / увеличить] + [метрика] + [объект] + [контекст]`.
Эталон: не «ускоряет работу», а, например, «сократить время руководителя на поиск ошибки в одном документе» или «увеличить долю заказов, собранных без переделки». Не сводится к формуле → помечай `[BUZZWORD]`. _^[inferred — правка Ulwick]_

**4. Все выводы — только через status quo клиента** (его текущий способ: таблица в Excel, ручной контроль, делать ничего / «и так сойдёт»). **Запрещено** «у нас vs у них». Формула вывода: **«их незакрытая работа клиента = наш потенциальный job»**.

**5. Любая ЦИФРА (выручка / число клиентов / раунд / % эффекта) — кандидат на ручную верификацию.** Модель в режиме поиска часто ошибается с атрибуцией источника (особенно на цифрах) — «есть ссылка» ≠ «факт верный». Каждую цифру помечай `[ЦИФРА — verify]` и перед сводом открой ссылку руками: ✅verified / ⚠️not-found / ⚠️ссылка-не-подтверждает. Невыверенная цифра в финал не идёт — уходит в гипотезы.

> Подробнее про линзу — в [04-ajtbd-lens.md](../methodology/04-ajtbd-lens.md). Про сигналы и сходимость — в [02-signal-rules.md](../methodology/02-signal-rules.md).

## P-A — Позиционирование + наблюдаемая Core Job + проверка артефактом

Цель: вытащить, что конкурент обещает, и проверить, стоит ли за обещанием реальная работа клиента (или это лозунг).

```
Используй режим Deep Research. Каждое утверждение — источник + дата (приоритет последних 2–3 лет). Размечай: [САЙТ/продавец] vs [ГОЛОС КЛИЕНТА].

Контекст. Конкурентная разведка по {{КОНКУРЕНТ}} ({{САЙТ_URL}}) через Advanced JTBD (Jobs To Be Done). Ниша — {{НИША}}. Нужна РАБОТА (Job), которую решение реально убирает у клиента, а не их маркетинг.
Источники: главная {{САЙТ_URL}}, страницы «о продукте / возможности», заголовки H1/H2.

Собери:
1. One-liner / pitch дословно + URL.
2. Глаголы ценности из заголовков как ГИПОТЕЗУ уровня работы:
   • Big (обещание-обёртка: «расти», «зарабатывать больше», «масштабироваться»);
   • Core (рабочая суть: «контролировать», «видеть картину», «держать под контролем»);
   • Small / commodity (узкая операция: «оформить», «посчитать», «расшифровать»).
3. ПРОВЕРКА уровня артефактом: для каждого Big-обещания найди доказательство (named-кейс с атрибуцией, цифра с описанным методом замера). Big без артефакта → [ПСЕВДО-BIG] и понизь до фактической работы, которую решение убирает.
4. НАБЛЮДАЕМОЕ (не ярлык): какую работу клиента решение УБИРАЕТ полностью (клиент перестаёт делать X руками), какую СОКРАЩАЕТ, какую перекладывает на себя — на каждую цитата.
5. Если на сайте только Big-формулировки без Core — это сигнал (продают на эмоции, Core скрыт): реконструируй вероятный Core Job из описания функций, пометь ^[inferred].

Сравнивай не со мной, а со status quo клиента (его текущий способ: ручной учёт, таблица, «делать как раньше», «ничего не делать»).
Формат: нумерованный список, каждый пункт = факт + цитата + URL + дата.
Финал (1 строка): фактический уровень их Core Job + чем он отличается от status quo клиента.
```
🌐 EN-switch: переведи нишу и глаголы на английский (grow / control / record); вопрос 3 → «is the Core Job above / at / below '[короткая формулировка их рабочей сути]'».

## P-B — ICP / сегменты + язык клиента + fast/slow воронка

Цель: понять, КОМУ продают, НА КАКОМ ЯЗЫКЕ и где границы их идеального клиента (ICP).

```
Используй Deep Research. Источник + дата (последние 2–3 года). Размечай: [САЙТ/продавец] vs [ГОЛОС КЛИЕНТА].

Контекст. Разведка по {{КОНКУРЕНТ}} ({{САЙТ_URL}}) через Advanced JTBD. Ниша — {{НИША}}. Нужно КОМУ продают и НА КАКОМ ЯЗЫКЕ — границы их ICP (идеального клиента).
Источники: сегментные / отраслевые лендинги, «кейсы / клиенты», логотипы, отзывы, тарифы (прокси размера компании).

Собери:
1. Роли ЛПР (лица, принимающего решение) — по кому написаны заголовки и кейсы. Примеры из разных сфер: руководитель / собственник, операционный директор, закупщик, главный технолог, маркетолог, владелец точки.
2. Размер компаний (число сотрудников, объём операций, выручка, крупный бизнес vs малый) — по логотипам и цифрам кейсов.
3. Отрасли — где сильны, где явно отсутствуют.
4. Границы ICP: для кого ЯВНО не предназначено («от N сотрудников», «для крупных команд», «только для розницы» и т.п.).
5. ПРОПОРЦИЯ языка: личная работа конкретного человека («перестанете делать это руками», «вы получите контроль») vs бизнес-работа организации («конверсия +X%», «выручка», «маржа») — примерное соотношение в % с цитатами. Это маркер реального ICP.
6. Боль клиента — РАЗДЕЛИ: [ИХ ФОРМУЛИРОВКА] (копирайтер) vs [ГОЛОС КЛИЕНТА] (цитата из кейса / отзыва). Ценны только вторые.
7. Заточка воронки fast vs slow: self-serve / открытая цена / «начните сейчас» = fast; демо / консультация / расчёт окупаемости / white paper = slow. Если только fast → slow-сегмент (кто принимает решение долго и взвешенно) свободен = ниша; если только slow → наоборот.
8. Для каждого сегмента — какой Core Job доминирует и где он over- / under- / appropriately-served. importance (важность для сегмента) и satisfaction (как закрыто) оценивай ДВУМЯ независимыми осями, не одним ярлыком: over = низкая важность × высокая удовлетворённость, under = высокая важность × низкая удовлетворённость, appropriately = совпадают (не трогать). Один и тот же Core Job может быть under-served для одного сегмента и over- для другого — давай разрез по сегменту, не схлопывай.

Формат: таблица «сегмент | роль ЛПР | размер | отрасль | fast/slow | цитата + URL + дата».
Финал (2 строки): ядро их ICP + сегмент, который они НЕ покрывают.
```
🌐 EN-switch: роли → переведи под западный рынок (например VP / Director / Head of …); источники → отраслевые каталоги и обзорные площадки (G2 / Capterra и аналоги для вашей ниши).

## P-C — Триггеры + 4 силы (перевод продавец → клиент) + альтернативы

Цель: восстановить 4 силы переключения, триггеры смены поведения и альтернативы — включая невидимые.

```
Используй Deep Research. Источник + дата (последние 2–3 года). Размечай: [САЙТ/продавец] vs [ГОЛОС КЛИЕНТА].

Контекст. Разведка по {{КОНКУРЕНТ}} ({{САЙТ_URL}}) через Advanced JTBD. Ниша — {{НИША}}. Нужны 4 силы переключения, триггеры и АЛЬТЕРНАТИВЫ (включая невидимые). ВАЖНО: сайт — отполированный pull; push / anxiety / habit на нём почти отсутствуют, их надо РЕКОНСТРУИРОВАТЬ и честно помечать ^[inferred].
Источники: кейсы, лендинги «почему мы / проблема», FAQ, страницы возражений, отзывы.

Собери:
1. 4 силы. Для КАЖДОЙ укажи источник: [ЦИТАТА с сайта] / [РЕКОНСТРУКЦИЯ ^inferred] / [НЕ НАЙДЕНО].
   • PUSH (проблемы старого способа), PULL (ценность нового), ANXIETY (страхи внедрения — цена, интеграция, «сотрудники будут против», «не разберёмся»), HABIT (привычка к статус-кво).
   Если ставишь push / боль из лозунга («теряете деньги на ошибках») — это ПРОДАВЦОВСКИЙ push: пометь [ПРОДАВЕЦ] и рядом переведи в КЛИЕНТСКИЙ — кто конкретно (роль) и его личная ставка (премия / репутация перед руководством / страх не выполнить план / горящий дедлайн).
2. Struggling moment (момент борьбы) — оцени по 6 признакам (место / время / должность / событие / эмоция / дата), балл X/6. На сайте ожидаемо 0–1/6 — это норма, отметь «не раскрыт» и дай реконструкцию.
3a. Активирующее ЗНАНИЕ: какой факт клиент должен ПОНЯТЬ, чтобы вообще начать рассматривать такое решение (например: «эту проверку можно автоматизировать, а не делать руками»; «инструмент не ошибётся чаще человека»). Где на сайте дают это знание (онбординг / FAQ / разъясняющая статья)? Если нет — продают на общем pull, слабее = окно для входа.
3b. Активирующее СОБЫТИЕ (часть триггера, НЕ путать с 3a): конкретный факт / цифра, вскрывающая скрытую проблему (например: «30% заказов уходят с ошибкой в спецификации»).
4. Видимые альтернативы — с кем прямо сравнивают.
5. Невидимые альтернативы (главные, обязательные кандидаты): таблица в Excel, ручная проверка силами сотрудника, обучение / тренинги, наём подрядчика-аудитора, «ничего не делать» — что из этого они атакуют, а что игнорируют.

Сравнивай со status quo клиента, не со мной.
Формат: блок на каждую силу (с колонкой продавец → клиент), блок «триггер X/6», таблица альтернатив.
Финал (1 строка): какую невидимую альтернативу они НЕ закрывают.
```
🌐 EN-switch: push / pull / anxiety / habit оставь как есть; невидимые альтернативы → spreadsheets / manual review / training / hiring a contractor / doing nothing.

## P-D — Ценность как outcome + что убирает + 3 энергии + упаковка/aha

Цель: перевести ценность из «фич вендора» в измеримый результат клиента и разобрать, как они упаковывают момент «вау».

```
Используй Deep Research. Источник + дата (последние 2–3 года). Размечай: [САЙТ/продавец] vs [ГОЛОС КЛИЕНТА].

Контекст. Разведка по {{КОНКУРЕНТ}} ({{САЙТ_URL}}) через Advanced JTBD. Ниша — {{НИША}}. Нужна ценность как ИЗМЕРИМЫЙ outcome клиента (не фича вендора), что решение убирает, и упаковка момента «вау» (aha).
Источники: главная, тарифы / pricing, демо / онбординг, кейсы с цифрами.

Собери:
1. Цифры ROI → REALITY-CHECK: чьи это цифры, на какой выборке, каким методом замерены. Сбей оптимизм поправками: заявленный эффект делится на 2–3 (selection bias — показывают лучшие кейсы), минус задержка на внедрение, минус регрессия к среднему. Необоснованные цифры помечай [НЕОБОСНОВАНО]. Любую конкретную цифру (эффект / выручка / число клиентов) дополнительно помечай [ЦИФРА — verify] — её перед сводом открывают и сверяют руками (см. правило 5). (Не вшивай готовые «нормальные коридоры» — выводи реалистичную оценку из самих данных кейса.)
2. КАЖДОЕ обещание → desired outcome клиента: [сократить / увеличить] + [метрика] + [объект] + [контекст]. Эталон: не «ускоряет работу», а «сократить время руководителя на поиск ошибки в одном документе». Не сводится → [BUZZWORD].
3. Энергоэффективность по 3 энергиям:
   • функциональная (часы / деньги / усилия),
   • социальная (что подумает команда, не будет ли сопротивления сотрудников),
   • эмоциональная (страх ЛПР облажаться перед руководством / клиентами).
   Какую энергию адресуют, какую игнорируют — игнорируемая = окно для входа.
4. НАБЛЮДАЕМОЕ (не номер механики): какую работу клиента решение убирает полностью / сокращает / поднимает на уровень выше (решает работу-родителя, делая под-работы ненужными) — на каждую цитата.
5. Time-to-value: за сколько времени клиент видит первый результат. Какая его метрика сдвигается ПЕРВОЙ.
6. Упаковка / aha: визуальные приёмы (дашборд, скоринг, демо-отчёт, наглядное «до / после»), через что показывают ценность за 5 секунд.

Buzzword-фильтр строго: «качество / эффективность / лучший» без числа → [BUZZWORD].
Формат: таблица «обещание | outcome клиента | цифра | обоснованность | какая энергия».
Финал (1 строка): сильнейший рычаг ценности конкурента.
```
🌐 EN-switch: 3 энергии → functional / social / emotional; остальное переведи под рынок.

## P-E — Слабые места + over/under-served + job map покрытия + вектор

Цель: вытащить реальные слабости (из голоса клиента), разложить работу клиента на шаги и понять, куда конкурент движется.

```
Используй Deep Research. Источник + дата (последние 2–3 года). В отзывах — это [ГОЛОС КЛИЕНТА], главный источник правды в наборе.

Контекст. Разведка по {{КОНКУРЕНТ}} ({{САЙТ_URL}}) через Advanced JTBD. Ниша — {{НИША}}. Нужны РЕАЛЬНЫЕ слабые места, непокрытые шаги работы клиента и вектор (куда они идут).
Источники: отзывы (обзорные площадки, отраслевые форумы, соцсети), блог, вакансии (карьерные страницы, job-площадки), новости / релизы.

Собери:
1. Частые ЖАЛОБЫ (цена, сложность, точность, поддержка, онбординг) — реальные цитаты + источник + дата.
2. Из НЕГАТИВНЫХ отзывов восстанови несработавшее переключение: что клиент ожидал (pull) → на что напоролся (новый push) → почему вернулся к старому (habit победил). Это карта сил, на которой конкурент проигрывает = свободный сегмент.
3. OVER- / UNDER-served — importance и satisfaction оцениваются как ДВЕ НЕЗАВИСИМЫЕ оси (не один ярлык). Для каждого outcome / шага: важность для клиента (низкая / средняя / высокая) ОТДЕЛЬНО от удовлетворённости тем, как закрыто (низкая / средняя / высокая). Классификация по пересечению:
   • over-served = важность низкая × удовлетворённость высокая (вложено больше, чем нужно: «слишком сложно / переплачиваю / не используем половину») → ниша для упрощения / удешевления;
   • under-served = важность высокая × удовлетворённость низкая → точка входа;
   • appropriately-served = важность и удовлетворённость совпадают (закрыто ровно как надо) → НЕ ТРОГАТЬ, помечай явно, чтобы не выдать за дыру.
   ВАЖНО: один и тот же outcome может быть over-served для одного сегмента (роль / размер компании) и under-served для другого — не схлопывай в общий ярлык, давай разрез по сегменту.
   Размер «дыры» меряй не бинарно «есть/нет жалоб», а размером и сегментом спроса: для какой доли / какого разреза (роль, размер бизнеса, отрасль) этот outcome важен и не закрыт.
   ЧЕСТНЫЕ МЕТРИКИ (см. правило 5 и [02-signal-rules.md]): НЕ выдавай долю числом без определённого знаменателя. Вместо «30% жалуются» пиши «повторяющаяся тема в N независимых отзывах с цитатой за период [даты]». Один отзыв = шум; повторяемость в нескольких независимых источниках = сигнал.
   REALITY-CHECK на under-served: прежде чем объявить outcome недообслуженным, сбей оптимизм — отделяй «реально болит и важно» от «громко жалуется один сегмент» и от разовых негативных всплесков; заявленный масштаб дыры дели на /2–3, как и ROI-цифры (selection bias отзывов — недовольные пишут чаще). Дыра засчитывается, только если выдержала эту поправку.
4. JOB MAP покрытия — ТРИ ОТДЕЛЬНЫЕ КОЛОНКИ на каждый шаг (не схлопывать в одну оценку): importance клиента | satisfaction клиента | покрытие конкурентом (0/1/2). Разложи работу клиента на шаги. Пример (адаптируй под {{НИША}}): [понять, что именно проверять / делать → найти проблемные места → разобрать причину → внести исправление / дать обратную связь исполнителю → проверить, что исправлено → отследить динамику во времени]. Точка входа = шаг с высокой importance × низкой satisfaction × низким покрытием конкурента (часто «петля доведения» — довести исполнителя до фактического исправления). Шаг с высоким покрытием при высокой satisfaction помечай appropriately-served — он не дыра.
5. SIGNAL CONVERGENCE (вектор): собери 3+ НЕЗАВИСИМЫХ сигнала и проверь, сходятся ли они:
   • новые роли-персонажи в свежих кейсах (миграция ВВЕРХ — к стратегии / выручке);
   • интеграции с инструментами для малого бизнеса / снижение порога входа (атака на НИЖНИЙ сегмент);
   • найм (вакансии: enterprise-продажи? разработка ML? партнёрства?);
   • темы блога за последний год.
   Вывод делай только если 3+ сигнала сходятся, иначе пиши «сигналы расходятся».

Запрещено «у нас vs у них» — только через незакрытую работу клиента.
Формат: блок «жалобы», блок «несработавшие переключения», таблица «шаг job map | importance | satisfaction | покрытие 0/1/2 | класс (over / under / appropriately-served) | сегмент», таблица «сигнал | направление | источник + дата».
Финал: список «их незакрытый шаг / работа → наш потенциальный job» (3–5 пунктов) — только under-served шаги, выдержавшие reality-check; appropriately-served в финал НЕ выноси.
```
🌐 EN-switch: отзывы → обзорные площадки и Reddit; вакансии → LinkedIn Jobs и аналоги; пункты 4–5 — по той же логике.

## Как пользоваться

1. Подставьте `{{КОНКУРЕНТ}}`, `{{САЙТ_URL}}`, `{{НИША}}`. Для западного конкурента — переключитесь на английский по пометкам 🌐 EN-switch.
2. Прогоняйте по очереди **P-A → P-E** в режиме Deep Research. Если времени мало — приоритет на **P-C** (силы + альтернативы) и **P-E** (слабости + вектор): они дают больше всего точек для отстройки.
3. Ответы разносите по блокам [досье](../examples/gong/dossier-skeleton.md). Маркеры `[САЙТ/продавец]` / `[ГОЛОС КЛИЕНТА]` / `[ПСЕВДО-BIG]` / `[BUZZWORD]` / `[НЕ НАЙДЕНО]` / `[ЦИФРА — verify]` / `appropriately-served` **не стирайте** — это и есть карта слабых мест конкурента (и карта того, что трогать НЕ надо). Перед сводом: каждую `[ЦИФРА — verify]` откройте руками и проставьте ✅verified / ⚠️not-found / ⚠️ссылка-не-подтверждает; невыверенные цифры в финал не выносите.
4. Итог по конкуренту → строкой в [battlecard](../templates/battlecard.md) + новые приёмы упаковки в [examples/packaging-tactics.md](../examples/packaging-tactics.md).

> ⚠️ Промпты намеренно плотнее «голых 2 тыс. символов» — это цена точности (разметка источника, перевод продавец → клиент, job map). Если Perplexity обрезает ответ — режьте нумерованные пункты с конца, а ядро (пункты 1–3) держите.

## Доп. deep-dive промпты (P-F, P-G) — опциональный слой

> Аддитивны к ядру P-A…P-E (наследуют те же 4 правила разметки). Закрывают две грани, которых нет в ядре: **монетизацию** и **switch-timeline** (путь переключения клиента во времени). Гоняйте опционально, для глубокого разбора.

### P-F — Монетизация / упаковка тарифов через AJTBD

```
Используй Deep Research. Источник + дата (последние 2–3 года). Размечай: [САЙТ/продавец] vs [ГОЛОС КЛИЕНТА].

Контекст. Разведка по {{КОНКУРЕНТ}} ({{САЙТ_URL}}) через Advanced JTBD. Ниша — {{НИША}}. Нужна МОДЕЛЬ МОНЕТИЗАЦИИ и где ценовой барьер, а не сам прайс.
Источники: страница тарифов / pricing, FAQ по оплате, отзывы про цену.

Собери:
1. Модель: подписка / за место (seat) / за объём (операции, минуты, штуки) / разовая. Что является «счётчиком» цены — за что именно платят.
2. Упаковка тарифов: какие тарифы, и КАКУЮ РАБОТУ клиента открывает каждый следующий (не «фичи тарифа», а что клиент НАЧИНАЕТ МОЧЬ делать). Что прячут в верхний тариф / «по запросу».
3. Якорь и апселл: с чего начинают цену, чем поднимают чек (доп. места, модули, интеграции).
4. Ценовой барьер как ANXIETY (не просто «дорого»): для какой роли ЛПР цена — личная ставка (премия / бюджет / риск перед руководством). [ГОЛОС КЛИЕНТА] из отзывов про цену важнее [САЙТ].
5. Скрытая стоимость: контракт (срок, автопролонгация, штраф), порог входа (внедрение, обучение), что в отзывах зовут «переплатой / не используем половину».
6. Status quo сравнение: их прайс vs «ничего не делать» (0) vs ручной способ (зарплата сотрудника, который делает это руками) — чем оправдывают разницу.

Buzzword-фильтр: «гибкие тарифы / по запросу» без цифр → [НЕПРОЗРАЧНО].
Формат: модель + таблица «тариф | какую работу открывает | цена | барьер/anxiety | цитата + URL».
Финал (1 строка): где монетизация создаёт незакрытую работу / барьер = окно для входа (например: «дорого для небольших команд», «платишь за то, чего не используешь»).
```
🌐 EN-switch: источники → pricing-обзоры на обзорных площадках; seat / usage / flat — как есть.

### P-G — Switch-timeline + заточка GTM-motion

```
Используй Deep Research. Источник + дата (последние 2–3 года). Размечай: [САЙТ/продавец] vs [ГОЛОС КЛИЕНТА]. ВАЖНО: timeline на сайте не дан — РЕКОНСТРУИРУЙ из кейсов / отзывов и честно помечай ^[inferred].

Контекст. Разведка по {{КОНКУРЕНТ}} ({{САЙТ_URL}}) через Advanced JTBD (forces, Moesta). Ниша — {{НИША}}. Нужен ПУТЬ переключения клиента и под какую его стадию заточен заход конкурента.
Источники: кейсы (особенно «как пришли к решению»), отзывы, лендинги «почему мы», демо / триал-страницы, FAQ.

Собери:
1. Switch-timeline по стадиям Moesta — для каждой укажи, что её запускает у клиента + есть ли свидетельство [ГОЛОС КЛИЕНТА] из кейса/отзыва или это реконструкция ^[inferred]:
   • First Thought (первая мысль «надо менять») — какое событие;
   • Passive Looking (присматривается) — что читает / гуглит;
   • Active Looking (активно сравнивает) — какие критерии сопоставляет;
   • Deciding (решается) — что добивает (демо / пилот / кейс / цена);
   • Onboarding (первые недели) — что должно сработать, чтобы клиент не откатился назад.
2. Под какую стадию заточен ИХ заход (по контенту / CTA): self-serve + открытая цена + «начать сейчас» = ловят на Active / Deciding (fast); демо + расчёт окупаемости + white paper + «оставьте заявку» = тянут с Passive (slow).
3. Где в timeline ДЫРА: какая стадия не обслужена (нет сопровождения на старте → откат на Onboarding; нет контента для First Thought → не ловят тех, кто ещё не осознал проблему).
4. Кто на каждой стадии РЕШАЕТ / ВЛИЯЕТ (роль ЛПР) и его личная ставка.

Сравнивай со status quo клиента, не со мной.
Формат: таблица «стадия | триггер | свидетельство [голос]/[inferred] | их заточка | дыра».
Финал (1 строка): стадия timeline, где конкурент слаб = окно для входа.
```
🌐 EN-switch: стадии Moesta — EN как есть (First Thought / Passive Looking / Active Looking / Deciding / Onboarding); источники → case studies на обзорных площадках.

> ⚠️ P-F / P-G не проходили арену 3 школ, как ядро P-A…P-E. Обкатайте на живом конкуренте, прежде чем считать каноном (особенно «какую работу открывает тариф» в P-F и определение стадий в P-G).
