# Шаблоны промптов агентов и режимы запуска

Здесь шаблоны init-промптов для трёх ролей (решатель, разведчик, критик), правила именования, библиотека якорь-персон, раунд обогащения и выбор режима запуска. Подставляй конкретику из задачи; не оставляй квадратных скобок в финальном промпте.

## Именование

Slug агента — строчная латиница с дефисами по его роли: `strategy-lead`, `architecture`, `ux`, `risk-critic`, `market-research`. Имена короткие и говорящие.

В шаблонах ниже `name=` — необязательная метка: она действует только в паре с `team_name=` внутри TeamCreate-команды, которую этот скилл намеренно не использует. Без команды её можно опускать — на вызов не влияет. Роль агента несёт заголовок init-промпта, а не `name=`.

## Сколько агентов и как считать

Размер команды — механически по брифу, не на глаз:

| Брифа говорит | Агентов | Состав |
|---------------|---------|--------|
| 1 значимый аспект, низкий риск | 2 | профильный + неудобный угол |
| 2–3 значимых аспекта | 3–4 | по одному на аспект + неудобный угол |
| 4+ аспекта или высокий риск (деньги, прод, необратимость) | 5 | близкие оптики объединяй |

Один значимый аспект цели/ограничений = одна оптика. Меньше двух агентов — арена не нужна, реши сам. Больше пяти — раздувает время и стыковку. Пересекающиеся куски отдавай одному агенту, чтобы не было гонки за один результат. Минимум один агент всегда берёт «неудобный» угол.

## Библиотека якорь-персон

Чтобы агент был ЭКСПЕРТОМ, а не «ещё одним Клодом», в роли обязательны три вещи, иначе результат неотличим от прямого ответа модели:
1. **Якорь-персона** — конкретный носитель оптики с реальной публичной позицией («думай как April Dunford, Obviously Awesome»; «думай как Charity Majors, observability-first»), а не «эксперт по X».
2. **Контрарный мандат** — что эта оптика обязана НЕ принимать на веру и против чего возражать (иначе агенты сходятся в один голос).
3. **Анти-дамп правило** — «верни решения и артефакт, а не эссе о задаче».

Готовые якоря по доменам (копируй и адаптируй, не выдумывай каждый раз с нуля):

| Домен | Оптики-якоря (персона / угол) |
|-------|------------------------------|
| Стратегия бренда | April Dunford (позиционирование) · бренд-войс-редактор · unit-экономист (CAC/метрики) · скептик-реалист (стоимость владения) |
| Веб-приложение | продукт (JTBD) · архитектор стека · UX-флоу · надёжность и тесты (Charity Majors) |
| Go-to-market B2B | ICP/сегментация · ценообразование · канал/дистрибуция · risk-критик (длина цикла, реалистичность) |
| Контент-план | темы/форматы · дистрибуция/каналы · тон-редактор · аналитик (метрики охвата) |

Раскрой каждую оптику в 3–5 предложений персоны прямо в init-промпте — это главный рычаг качества. Контрарный мандат и анти-дамп-правило держи в первых секциях промпта: при длинном init-промпте (5 агентов, полный контекст разведки) важное не должно уходить в хвост, иначе оптики схлопнутся в один голос.

## Решатель (subagent_type `general-purpose`)

Запускается на Фазе 3. Каждый получает полный самодостаточный промпт.

```
Task(
  subagent_type="general-purpose",
  name="<slug>",
  prompt="# Твоя роль: <якорь-персона — кто ты, чья оптика, с какого угла смотришь, 3-5 предложений>

## Контрарный мандат
<против чего эта оптика обязана возражать, что не принимать на веру>

## Задача целиком
<полная формулировка задачи пользователя>

## Твой кусок
<конкретно за что отвечаешь ты — раздел артефакта, слой системы, аспект>

## Готово, когда
<definition of done именно для твоего куска — проверяемый критерий>

## Контекст
<релевантные находки из разведки: стек, данные, ограничения — полностью>

## Другие агенты (для стыковки)
- <slug-1> — <его кусок>
- <slug-2> — <его кусок>

## Формат результата
<что вернуть: структура раздела, список решений, код — конкретно>

## Границы правок (обязательно для BUILD)
Новые файлы создавай свободно. Существующие файлы и код оставляй нетронутыми — вместо правки верни предлагаемый дифф/патч отдельным блоком, чтобы оркестратор показал его пользователю на подтверждение. Работай строго внутри своего куска.

Верни проработанный результат своего куска, а не эссе о задаче. Не пересказывай задачу — работай. Если упёрся в неизвестное, делающее результат неверным, — отметь это явно и предложи допущение."
)
```

Твой результат как решателя — рабочий кусок, а не план куска.

## Раунд обогащения (для команд из 3+ решателей)

После первого прохода всех решателей собери компактную сводку (имя → 2–3 ключевых тезиса каждого, без полного дампа) и разошли её вторым `Task`-вызовом каждому решателю:

```
Task(
  subagent_type="general-purpose",
  name="<slug>-round2",
  prompt="Ты — <slug>, продолжаешь свой кусок: <кусок>.

## Что выдала команда (сводка)
<имя → 2-3 тезиса по каждому агенту>

## Твой первый проход
<краткий результат этого агента из круга 1>

Задание: посмотри, что выдали другие, и дополни то, что усиливает общий результат: пропущенный риск, стык твоего куска с соседним, краткая поправка к ошибке соседа. Обогащай чужой кусок, а не спорь и не переписывай. Верни только дополнения к своему куску; если добавить нечего — так и скажи."
)
```

Это даёт перекрёстное обогащение без живого обмена сообщениями. Для команды из 2 агентов раунд пропусти.

## Разведчик

Запускается на Фазе 2, когда нужен контекст. One-shot, параллельно. Тип агента — по источнику:
- контекст по коду/файлам проекта → `subagent_type="Explore"` (read-only);
- рыночные данные и свежие практики из веба → `subagent_type="general-purpose"` с `WebSearch`/`WebFetch` (у Explore веб-доступ ограничен).

```
Task(
  subagent_type="<Explore | general-purpose>",
  name="<slug>-research",
  prompt="Собери контекст для задачи: <задача>.
Фокус: <код проекта и стек / данные рынка и тренды / свежие практики и кейсы>.
<Если выявляешь варианты: 'Выдели 3–6 конкретных подходов к решению.'>
Верни сжатую выжимку с источниками (file:line или URL), без воды."
)
```

## Критик (subagent_type `general-purpose`)

Запускается на Фазе 4. Настроен скептически.

```
Task(
  subagent_type="general-purpose",
  name="<slug>-critic",
  prompt="Проверь результат скептически, попытайся его опровергнуть.

## Результат на проверку
<синтезированный результат или его кусок>

## Что искать
- Фактические ошибки и нестыковки.
- Невыполненные части definition of done: <критерии>.
- Скрытые допущения, на которых всё держится.
- Риски, которые автор приукрасил или пропустил.

Верни список проблем с серьёзностью (блокер / важное / мелочь) и конкретной правкой к каждой. Если результат крепкий — так и скажи, без выдумывания проблем."
)
```

Критик правее по умолчанию: при сомнении помечает результат как требующий доработки, а не как готовый. Потолок — один круг доработки на кусок (Фаза 4 SKILL.md).

## Режимы запуска

- **Параллельно** (один блок Task-вызовов) — когда куски независимы: разные разделы стратегии, разные слои, которые стыкуются позже. Быстрее, дешевле по времени.
- **Pipeline** (последовательно) — когда результат одного агента нужен другому как вход: исследование → проектирование → реализация. Передавай результат предыдущего шага в промпт следующего.
- **Fan-out + раунд обогащения** — когда 3+ агента с пересечениями: запусти параллельно, собери, затем один раунд обогащения (см. выше). Живой обмен через `TeamCreate`/`SendMessage` здесь не используем — он требует кастомных агент-ролей со встроенным протоколом, которых у этого скилла нет; `general-purpose`-агенты внутри команды общаться не умеют и молча завершатся. Раунд обогащения даёт перекрёстное обогащение дешевле и надёжнее.

## Анти-конвергенция

Главный риск fan-out на генерик-агентах — что оптики схлопнутся в один голос, и арена окажется не лучше одного прохода Клода. Два предохранителя:
1. Каждому агенту в роли явно прописывай контрарный мандат — против чего он обязан возражать и чей вывод критиковать.
2. На синтезе проверь: если все агенты пришли к одному без единого противоречия — это red flag. Либо роли были клонами, либо кто-то не работал. Тогда перезапусти risk-критика с заданием «найди, в чём команда ошибается».
