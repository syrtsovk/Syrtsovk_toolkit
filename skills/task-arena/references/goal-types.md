# Типы цели — как задача определяет команду и выдачу

Task Arena универсальна: разные задачи требуют разной механики. Тип цели уже намечен в брифе (поле «Тип цели (предв.)») — здесь ты его подтверждаешь и ведёшь арену по нему, а не выводишь с нуля. Тип не жёсткий: задача может быть гибридной (например, «составь стратегию и сразу набросай лендинг» = CREATE + BUILD); тогда веди по доминирующему типу и добавь шаги второго.

## Матрица типов

| Тип | Сигналы в задаче | Состав команды | Разведка | Проверка | Формат выдачи |
|-----|------------------|----------------|----------|----------|----------------|
| **CREATE** — создать артефакт | «составь стратегию», «напиши план», «сделай ТЗ», «придумай концепцию», «разработай структуру» | Профильные эксперты, каждый прорабатывает свой раздел артефакта | Часто да — нужен контекст рынка/домена | Порогово — по важности артефакта | Markdown-документ в файл + резюме |
| **BUILD** — сделать работу в проекте | «запили фичу», «напиши код», «настрой», «собери приложение», «реализуй» | Исполнители по слоям (данные / логика / интерфейс / тесты) | Да — нужно прочитать код и стек | Да — результат исполняемый, риск выше | Новые файлы свободно; существующие — после «ок» |
| **ADVISE** — совет как лучше | «как лучше сделать X», «посоветуй подход», «какой стек выбрать», «с чего начать» | Эксперты с разными оптиками — дают рекомендации, не правят файлы | По необходимости | Порогово | Рекомендация в чате: вывод → почему → шаги |
| **DECIDE** — выбрать из вариантов | «что выбрать», «сравни A и B», «какой вариант лучше» | — | — | — | Если выбор ведёт к исполнению — продолжай как BUILD/CREATE после выбора. Если нужно чистое сравнение без исполнения — это работа expert-arena, передай туда |

## Как определять definition of done по типу

- **CREATE** — готов конкретный документ с оговорёнными разделами; пользователь может с ним работать без доработки.
- **BUILD** — изменение существует и работает (файлы созданы/правлены, при возможности — проверено запуском или тестом).
- **ADVISE** — пользователь получил однозначную рекомендацию с обоснованием и понимает первый шаг.
- **DECIDE** — выбор сделан и обоснован; если за выбором идёт исполнение, definition of done переопределяется под получившийся CREATE/BUILD. Граница тонкая: если из задачи неясно, нужно ли исполнение после выбора, — уточни это вопросом в Фазе 0 («только сравнить или сравнить и сделать?»), не угадывай. Чистое сравнение без исполнения уходит в expert-arena.

## Подбор экспертных оптик (для всех типов)

Бери реальные роли с разными углами, а не клонов:

- Для стратегии бренда: позиционирование / визуальная идентичность / голос бренда / экономика и метрики.
- Для приложения: продуктовый слой / архитектура и стек / UX / надёжность и тесты.
- Для контент-плана: темы и форматы / дистрибуция / тон и редактура / аналитика.

Минимум один агент берёт «неудобный» угол (риски, стоимость владения, реалистичность сроков) — ловит дыры, которые профильные оптимисты пропускают. Это Devil's Advocate арены, и у него есть зубы: каждое его возражение уровня «блокер» (результат не сработает / срок нереален / скрытая стоимость убивает идею) на Фазе 5 обязано быть либо закрыто в синтезе, либо вынесено пользователю явной строкой «Devil's Advocate: <возражение> — принимаем риск? [да / доработать]» через `AskUserQuestion`. Молча проглотить блокер неудобного угла нельзя — иначе арена выдаёт приукрашенный результат.

## Сквозной пример (CREATE)

Бриф: стратегия бренда кофейни, B2C, бюджет ограничен; готово = документ с позиционированием, визуалом, голосом и юнит-экономикой. Тип цели — CREATE, 4 значимых аспекта + риск бюджета → 5 агентов.

Команда: `positioning` (April Dunford), `visual-identity`, `brand-voice`, `unit-economics` (неудобный угол — метрики и стоимость), `risk-critic` (Фаза 4). Режим: решатели 1–4 параллельно одним блоком `Task`; раунд обогащения, т.к. 4 пересекающихся куска; критик после синтеза.

Синтез сшивает куски по структуре брифа, снимает противоречие «премиальный голос против ограниченного бюджета» (побеждает unit-economics — вопрос в его зоне), помечает выводы оптиками, сверяет таблицу definition of done и сохраняет в `docs/arena/YYYY-MM-DD-coffee-brand.md`.

## Сквозной пример (BUILD)

Бриф: добавить личный кабинет клиента в существующий сервис на Next.js; готово = страница отдаёт данные из API и покрыта хотя бы одним тестом, существующий код не сломан. Тип цели — BUILD, 3 аспекта + риск правки прода → 4 агента.

Команда: `data-layer` (API и запросы), `ui-flow` (экраны и состояния), `tests` (Charity Majors — надёжность), `risk-critic` (неудобный угол — что сломается в проде). Режим: сначала разведка кода (`Explore`), затем решатели **pipeline'ом** — UI зависит от формы данных, поэтому раунд обогащения не нужен. Новые файлы решатели создают свободно, правки существующих возвращают диффом на подтверждение. Критик обязателен (BUILD + существующие файлы). Definition of done зелёный только с пруфом: тест реально прошёл и его вывод приложен, а не «агент сказал, что сделал».

## Когда уступить другому скиллу

- Чистое продуктовое проектирование одной фичи через AJTBD → `product-architect`.
- Сравнение готовых вариантов без исполнения → `expert-arena`.
- Текст поста, КП, договор, презентация → профильные скиллы (`tgpost`, `kp-writer`, `dogovor`, `design-expert`).

Task Arena уместна, когда задача шире одного скилла и требует команды, доводящей её до результата.
