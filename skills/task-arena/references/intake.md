# Фаза прояснения — диагност и бриф

Эта фаза превращает любую формулировку, даже кривую и неполную, в чёткий бриф. Бриф — фундамент: на нём держится выбор специалистов и весь результат. Диагноста играешь ты сам, инлайн, без отдельного агента.

## Когда спрашивать, а когда нет

Решение принимай по полноте задачи, а не по привычке всегда уточнять:

| Состояние задачи | Действие |
|------------------|----------|
| Ввод покрывает 3+ оси из {цель, аудитория, ограничения, «готово»} — задача ясна | Вопросов не задавай — собери бриф и подтверди одной фразой |
| Понятна суть, но 1–2 критических пробела | Задай 1–2 точечных вопроса |
| Размытая, неполная или противоречивая («сделай мне стратегию», «запили что-нибудь крутое») | Задай 2–4 вопроса, по одному на каждый критический пробел |

Перед вопросами дёшево сними контекст сам (`Glob`/`Grep` по проекту или vault): есть ли уже целевой файл, папка или модуль. Не спрашивай то, что читается за один grep.

Спрашивай только то, без чего результат будет неверным. Остальное прими как разумное допущение и явно пометь в брифе — это снимает трение и не превращает прояснение в анкету. Потолок — 4 вопроса; дальше начинается friction.

## Что выяснить (критические пробелы)

1. **Цель и зачем** — что должно получиться и какую работу пользователя это решает.
2. **Для кого** — аудитория или потребитель результата.
3. **Ограничения** — сроки, бюджет, стек/инструменты, тон, обязательные требования.
4. **Объём и режим выдачи** — набросок, рабочий вариант или полная проработка; готовое решение/артефакт или совет «как лучше».
5. **Definition of done** — 1–2 проверяемых критерия, по которым видно, что задача закрыта.

## Как задавать вопросы

Используй `AskUserQuestion`. Один вопрос — один пробел. Каждый вопрос — блок с 2–4 конкретными вариантами под задачу плюс свободный ввод. Не печатай вопросы простым текстом — иначе пользователь отвечает расплывчато и кривой ввод не снимается. Между вопросами держи короткую строку-отражение (≤10 слов: «понял — это стратегия, уточню аудиторию»), чтобы не было ощущения допроса.

### Готовые блоки (подставь конкретику под задачу)

Цель и режим выдачи:
```json
{"questions":[{"header":"Что на выходе","question":"Что для тебя готовый результат по этой задаче?","multiSelect":false,"options":[{"label":"Набросок","description":"Черновик-направление, доработаю сам"},{"label":"Рабочий вариант","description":"Можно использовать, без полировки"},{"label":"Полная проработка","description":"Готовый артефакт под задачу"},{"label":"Совет, не артефакт","description":"Нужна рекомендация как лучше, не файл"}]}]}
```

Для кого:
```json
{"questions":[{"header":"Для кого","question":"Кто потребитель результата?","multiSelect":false,"options":[{"label":"<сегмент A под задачу>","description":"<кратко>"},{"label":"<сегмент B>","description":"<кратко>"},{"label":"Внутренняя команда","description":"Для себя/команды, не клиент"},{"label":"Своё","description":"Опишу текстом"}]}]}
```

Ограничения:
```json
{"questions":[{"header":"Ограничения","question":"Что обязательно учесть? (срок / бюджет / стек / тон)","multiSelect":true,"options":[{"label":"Жёсткий срок","description":"Назову дедлайн в ответе"},{"label":"Фикс. бюджет/стек","description":"Есть рамки по деньгам/инструментам"},{"label":"Тон/стиль","description":"Важен голос/формат"},{"label":"Ограничений нет","description":"Делай как считаешь лучше"}]}]}
```

Варианты формулируй под конкретную задачу, а не дословно из шаблона.

## Когда пользователь не помогает: «не знаю», «хватит», противоречие

- **«Не знаю / реши сам»** на любой вопрос → не зависай и не фантазируй: предложи 2–3 разумных направления, возьми одно как рабочее допущение и явно пометь его в брифе.
- **«Хватит, собирай»** → останови вопросы, собери бриф на том, что есть, остальное — в допущения.
- **Противоречие во вводе** (просят несовместимое — «дёшево и premium к завтрашнему дню») → не выбирай молча одну сторону. Верни противоречие пользователю одним вопросом: что приоритетнее. Если он отвечает «не знаю» — действуй по правилу выше.

## Шаблон брифа

Покажи пользователю перед проектированием команды:

```
## Бриф
**Задача:** [одна фраза]
**Зачем / результат для пользователя:** […]
**Для кого:** […]
**Ограничения:** […]
**Тип цели (предв.):** CREATE / BUILD / ADVISE / DECIDE — по goal-types.md
**Готово, когда:** [1–2 проверяемых критерия]
**Допущения:** [что принял без подтверждения]
```

Поле «Тип цели (предв.)» — мост в Фазу 1: тип уже выводится из собранного ввода, и Фаза 1 не определяет его заново с нуля. Бриф **не требует отдельного «ок»** — он уходит в единый гейт Фазы 1 вместе с составом команды, чтобы не было двух вопросов подряд. После показа брифа сразу проектируй команду.

## Пример: кривая задача → бриф

**Ввод:** «сделай мне стратегию бренда».

Критические пробелы: какой бренд и ниша, для кого, цель и срок, что считать готовым. Вопросы (по одному, через `AskUserQuestion` с вариантами):

1. «Какой бренд и ниша?» — варианты под контекст пользователя.
2. «Кто целевая аудитория?»
3. «Какая цель и горизонт?» — узнаваемость за 3 месяца / продажи / выход на новый сегмент.
4. «Что для тебя готовый результат?» — документ-стратегия / позиционирование + визуал / полный гайдлайн.

Из ответов собирается бриф (Тип цели: CREATE), и уже из него виден состав команды: позиционирование, визуальная идентичность, голос бренда, экономика и метрики, плюс неудобный угол — реалистичность бюджета.

## Вывод состава команды из брифа

Бриф прямо подсказывает специалистов: каждый значимый аспект цели и ограничений — кандидат в отдельную оптику. Definition of done подсказывает, кого добавить для проверки (например, агента по реалистичности сроков или стоимости). Дальше работа уходит в `goal-types.md` (тип цели) и `agent-prompts.md` (как написать роли и сколько агентов).
